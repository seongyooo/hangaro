## 왜 만드는가

**오버투어리즘 문제**: 경복궁, 북촌한옥마을 같은 유명 관광지에 인파가 집중되고, 덜 알려진 좋은 장소들은 비어 있다.

기존 카카오/네이버 지도는 "어떻게 갈지"는 알려주지만 "지금 어디가 덜 붐비는지"는 알려주지 않는다.

**HanGaRo의 포지션**: 혼잡도 정보를 시각화하고, 덜 붐비는 경로를 자연스럽게 제안한다.  
강제가 아닌 **정보 제공 + 대안 제시**를 통해 관광객 스스로 행동을 바꾸도록 유도한다.

## 핵심 가치 (무엇을 만들 것인가)

### 카카오/네이버 지도와 다른 점

| 기존 지도 | HanGaRo |
|-----------|---------|
| "어떻게 갈지" | "지금 어디가 덜 붐비는지" |
| 경로 안내 | 혼잡 회피 경로 추천 |
| 장소 정보 (영업시간, 메뉴 등) | 혼잡도 예측 + 버퍼 라우팅 |
| 단일 최적 경로 | Plan A/B/C 비교로 사용자 선택 |

### 만들지 않을 것

- 장소 상세 정보 (주소, 영업시간, 주변 카페 목록) → 카카오 따라가기
- 리뷰/평점 시스템
- 실시간 교통 정보

## 두 가지 핵심 시나리오

### 시나리오 1 — "어디 갈지 모르겠어"

```
지도 열기
  → 혼잡도 3D 막대로 어디가 붐비는지 한눈에 보임
  → "조용한 코스 추천받기" 탭
  → Plan A/B/C 비교 결과 즉시 표시
  → 사용자가 플랜 선택 후 이동
```

### 시나리오 2 — "경복궁 가고 싶어"

```
목적지 입력: 경복궁
  → 현재 혼잡도 판단 (0.88 → 혼잡)
  → 버퍼 라우팅 발동:
     "경복궁이 지금 매우 혼잡해요.
      낙산공원 먼저 → 2시간 후 경복궁 방문하면 혼잡도가 61% 줄어요"
  → 추천 코스로 바꾸기 / 원래대로 선택
```

시나리오 2가 **서비스의 존재 이유를 가장 잘 보여주는 장면**이다.  
공모전 심사에서도 이 흐름 하나가 전체를 설명한다.

## Plan A / B / C 알고리즘

| 플랜 | 이름 | 알고리즘 | 목적 |
|------|------|---------|------|
| **A** | 혼잡 최소 | Modified Dijkstra (혼잡도 가중치) | 인파를 최대한 피함 |
| **B** | 최단 경로 | Dijkstra (travel_min 기준, 혼잡도 무관) | 빠른 이동 |
| **C** | 숨은 명소 | hidden_gem_score greedy 선택 | 덜 알려진 명소로 자연 분산 |

### hidden_gem_score 계산

```python
gem = (rating / 5.0 × 0.4) + ((1 - congestion_avg) × 0.3) + (rank_score × 0.3)
```

- rating: KTO API 평점
- congestion_avg: 평균 혼잡도 (낮을수록 좋음)
- rank_score: 인기도 역순 (덜 알려질수록 높음)

### 버퍼 라우팅 (Buffer Routing)

혼잡 임계값(`congestion > 0.7`) 초과 시 발동:

```
목적지 혼잡 감지
  → TatsCnctrRateService로 혼잡 해소 예상 시간 계산
  → 해소까지 필요한 버퍼 시간만큼 근처 숨은 명소 경유
  → start → 숨은 명소 → 목적지 순 경로 반환
```

## 주요 설계 결정 기록

### 왜 카카오/네이버처럼 장소 상세 정보를 넣지 않는가

> "너무 많은 정보를 넣으려 하면 카카오/네이버 지도랑 다를 게 없어진다. 해당 지도보다 잘 만들 자신도 없고."

→ HanGaRo는 **혼잡도 회피**에만 집중. 장소 영업시간, 메뉴, 주변 카페 목록은 구현하지 않는다.

### 왜 버퍼 라우팅이 핵심인가

공모전 심사위원에게 서비스를 설명할 때, "경복궁 가고 싶은데 혼잡하네요 → 낙산공원 먼저 들르세요" 이 한 장면이 서비스 전체를 설명한다. 이 흐름이 제일 임팩트 있다.

### Plan A/B/C를 탭이 아닌 카드로 동시 표시하는 이유

탭으로 하면 비교가 안 된다. 3개를 동시에 보여주고 혼잡도 수치 차이를 눈으로 확인하게 해야 "Plan A가 실제로 혼잡을 줄여준다"는 메시지가 전달된다.

### 모바일 바텀시트 스냅 구현 이슈

React `onUp` 이벤트 핸들러에서 `sheetH` state를 읽으면 stale closure 문제가 발생한다 (pointerdown 시점의 값이 고정됨). `sheetDragRef.current.curH`에 실시간으로 저장하고 `onUp`에서 ref를 읽어 해결.

### Callout 카드 onClick 이벤트 버그

`buildCalloutEl`에서 `card`에 이미 `stopPropagation()`이 붙어 있어, 바깥 `wrap`에 리스너를 달면 이벤트가 막혀서 실행되지 않는다. `buildCalloutEl(pin, color, onClickCallback)` 형태로 콜백을 직접 card에 연결해서 해결.

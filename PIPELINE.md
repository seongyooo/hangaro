# HanGaRo — 전체 파이프라인 정의서

> **핵심 목표:** 오버투어리즘 방지  
> 사용자가 혼잡도를 인식하고, 덜 붐비는 경로를 자연스럽게 선택하도록 유도한다.  
> 강제가 아닌 **정보 제공 + 대안 제시**를 통한 행동 변화.

---

## 1. 문제 정의

| 현상 | 원인 | HanGaRo의 접근 |
|------|------|----------------|
| 유명 관광지에 인파 집중 | 정보 비대칭 — 사람들이 혼잡도를 모른 채 방문 | 실시간 혼잡도를 지도에 직관적으로 표시 |
| 비슷한 시간대에 몰림 | 동선 계획 수단 부재 | 혼잡 시간대를 피한 코스 자동 추천 |
| 숨은 명소 미발굴 | 검색 결과가 유명 장소 위주 | hidden_gem_score 기반 대안 장소 제안 |

## 2. 전체 파이프라인

```
[데이터 수집]
  KTO OpenAPI (areaBasedList1)  ←  관광지 목록, 좌표, 카테고리
  TatsCnctrRateService (KT ML)  ←  향후 30일 집중률 예측 (0~100)
  정적 패턴 폴백 (JSON)          ←  API 미등록 / 장애 시

       ↓ 정규화 (0.0~1.0)

[혼잡도 산출]
  congestion_service.get_congestion()
  - 1차: Redis 캐시 (TTL: CONGESTION_TTL)
  - 2차: KT TatsCnctrRateService API (cnctrRate / 100)
  - 3차: congestion_patterns.json 정적 패턴 (카테고리 × 요일 × 시간대)

       ↓ 라벨링

  0.0~0.3 → 한적(quiet)   초록
  0.3~0.5 → 여유(relaxed) 노랑
  0.5~0.8 → 보통(moderate) 주황
  0.8~1.0 → 혼잡(crowded) 빨강

       ↓ 경로 계획

[경로 최적화 (NetworkX 그래프)]
  build_graph(spots)  →  노드: 관광지, 엣지: 이동시간/혼잡도 가중치

  [모드 A] 자유 추천: modified_dijkstra → 혼잡도 낮은 경로 우선
  [모드 B] 목적지 고정: fixed_destination_route
    - 목적지 혼잡도 > 0.7이면 버퍼 라우팅 발동
    - predict_clear_time() → 혼잡 해소 예상 시각 계산
    - find_buffer_nodes() → 근처 숨은 명소로 시간 채움

       ↓

[API 응답]
  RouteResponse { spots[], total_congestion_avg, congestion_reduction_pct, message }

       ↓

[프론트엔드 표시]
  지도: 혼잡도 색상 펄스 노드 + 경로 폴리라인
  리스트: 장소명 / 혼잡 라벨 / 체류시간 / 이동수단
  배지: "혼잡도 -XX% 절감" / "평균 혼잡도 0.XX"
  버퍼 알림: "OO은 지금 붐빕니다. 근처 OO을 먼저 들러보세요."
```

## 3. 사용자 행동 흐름 (UX 파이프라인)

### 화면 1 — 메인 지도 (진입 즉시)

**사용자가 보는 것:**
- 현재 GPS 위치 중심의 지도 (서울 선로딩 없음)
- 주변 관광지에 혼잡도 색상 펄스 노드 자동 표시
- 하단 Bottom Sheet: 출발지 / 목적지 / 경유지 입력 + 이동수단 탭

**사용자가 할 수 있는 것:**
| 선택지 | 동작 |
|--------|------|
| 목적지 + 경유지 직접 입력 | 선택한 장소로 경로 구성 (API 불필요) |
| 목적지 없이 버튼 누름 | 현재 위치 기반 API 추천 코스 생성 |

**혼잡도 인식 포인트:**
- 지도 위 펄스 노드의 색상(초록→빨강)으로 직관적 인지
- 노드 탭 시 장소명 + 혼잡 라벨 툴팁 표시

---

### 화면 2 — 경로 탐색 중

**사용자가 보는 것:**
- 로딩 애니메이션 + 단계 텍스트 ("관광지 데이터 수집 중..." → "혼잡도 분석 중..." → "최적 경로 계산 중...")
- 최소 로딩 시간 보장 (빠른 응답도 자연스럽게 전환)

**목적:**
- 서비스가 단순 검색이 아닌 **혼잡도 기반 분석**임을 인식시킴

---

### 화면 3 — 결과 화면

**사용자가 보는 것:**

```
┌─────────────────────────────────┐
│  지도: 경로 폴리라인 + 혼잡 노드  │
├─────────────────────────────────┤
│  [플랜 A] [플랜 B] [플랜 C]      │
│                                 │
│  혼잡도 -61%    평균 혼잡도 0.28  │
│                                 │
│  ⚠ OO은 지금 붐빕니다.           │  ← 버퍼 알림 (혼잡 시에만)
│    [이 코스로 변경] [원래대로]    │
│                                 │
│  1. 창덕궁        [한적]  40min  │
│     └── 도보 15min               │
│  2. 낙산공원      [한적]  25min  │
│     └── 대중교통 10min            │
│  3. 이화마을      [여유]  35min  │
└─────────────────────────────────┘
```

**사용자가 선택하는 것:**

| 플랜 | 최적화 기준 | 혼잡도 영향 |
|------|------------|-------------|
| A — 혼잡 최소 | congestion 가중치 최대 | 인파를 최대한 피함 |
| B — 거리 최단 | 이동시간 최소 | 혼잡도 무관, 빠른 이동 |
| C — 숨은 명소 | hidden_gem_score 최대 | 덜 알려진 장소 → 자연 분산 |

**혼잡도 인식 포인트:**
- "혼잡도 -XX% 절감" 배지 → 수치로 명확하게 제시
- 각 장소별 혼잡 라벨 → 방문 전 예측 가능
- 버퍼 라우팅 안내 → 사용자가 직접 선택하도록 유도 (강제 아님)
- 드래그로 순서 변경 가능 → 자율성 보장

## 4. 혼잡도 데이터 흐름 상세

```
KTO TatsCnctrRateService API
  └─ 입력: area_cd, sigungu_cd, tats_nm (관광지명)
  └─ 출력: baseYmd × cnctrRate (집중률, 0~100)
  └─ 의미: 해당 날짜에 이 관광지에 얼마나 사람이 집중되는가

정규화
  cnctrRate / 100  →  congestion score (0.0~1.0)

라벨링
  0.0~0.3  한적  → 추천 대상 (우선 선택)
  0.3~0.5  여유  → 추천 가능
  0.5~0.8  보통  → 경고 없이 포함
  0.8~1.0  혼잡  → 버퍼 라우팅 발동 / 경고 표시

hidden_gem_score 산출
  = (rating/5.0 × 0.4) + ((1-congestion_avg) × 0.3) + (rank_score × 0.3)
  → 평점 높고, 평소 한적하고, 덜 알려진 장소일수록 높음
  → 플랜 C의 핵심 정렬 기준
```

## 7. 핵심 가치 명시 (심사 관점)

> **단순 길찾기 앱이 아닌, 오버투어리즘 해결 도구**

- **실시간 혼잡도 시각화** → 사용자가 붐비는 시간/장소를 미리 알 수 있음
- **혼잡도 기반 경로 최적화** → 인파를 자연스럽게 분산
- **버퍼 라우팅** → 혼잡 관광지 방문 전 시간을 숨은 명소로 채워 실질적 분산 효과
- **hidden_gem_score** → 알려지지 않은 대안 장소를 발굴해 유명 관광지 의존도 감소
- **강제가 아닌 선택** → 사용자 자율성 유지하면서 행동 변화 유도

- 실제 cctv를 연동해서 도로 cctv나 근처 명소 관련 cctv도 실제로 보게 해서 하는 방식은 어때?
